# PDF document policy

This policy governs PDFs and document-like files as domain records, not only as
binary payloads.

## Taxonomy

| Kind | Meaning | Current flows |
| --- | --- | --- |
| `pdf_export` | Generated report downloaded from a current data snapshot. It is not evidence by itself. | Auditoria, habilitacoes, relatorio de treinamentos do tripulante. |
| `pdf_evidence` | Uploaded or stored document that supports a domain fact. It requires metadata/blob consistency and retention policy. | Anexos de treinamentos and documentos de tripulante. |
| `signed_document` | Document whose evidential value depends on signer identity, integrity, and signing timestamp. | Reserved; no active signed flow is accepted as signed without these fields. |
| `temporary_document` | Generated document for operational consumption that is not persisted as evidence. | Manual do usuario generated on request. |

## Cross-cutting rules

- PDF payloads generated by the system must be validated as bytes and must start with the PDF magic bytes before an HTTP response is emitted.
- Generated PDF responses must use `application/pdf`, `no-store`, `nosniff`, and explicit `X-Document-*` headers.
- The original filename is display metadata only. It does not define identity, storage path, evidence value, or versioning.
- Evidence PDFs follow the upload, persistence, access, retention, and deletion policies already defined for files.
- A signed document is not inferred from extension, MIME type, or filename. It must have explicit signature semantics.

## Domain rules

| Domain | Kind | Storage | Versioning | Retention | Signature semantics |
| --- | --- | --- | --- | --- | --- |
| `auditoria.eventos` | `pdf_export` | `temporary_response` | `rendered_snapshot_at_request` | `not_persisted` | `unsigned_system_generated` |
| `relatorios.habilitacoes` | `pdf_export` | `temporary_response` | `rendered_snapshot_at_request` | `not_persisted` | `unsigned_system_generated` |
| `relatorios.tripulante.treinamentos` | `pdf_export` | `temporary_response` | `rendered_snapshot_at_request` | `not_persisted` | `unsigned_system_generated` |
| `manual.usuario` | `temporary_document` | `temporary_response` | `rendered_snapshot_at_request` | `not_persisted` | `unsigned_system_generated` |
| `treinamentos.anexos` | `pdf_evidence` | `persistent_document_storage` | `append_only_with_soft_delete` | `retained_until_governed_delete` | `unsigned_uploaded_evidence_hash_tracked` |
| `tripulantes.file` | `pdf_evidence` | `persistent_document_storage` | `replace_marks_previous_as_substituido` | `retained_for_traceability` | `unsigned_uploaded_evidence_hash_tracked` |

## Contract boundaries

- Data contract: filters, rows, summary, entity ids, and actor/request context used to generate the document.
- Rendering: ReportLab template and layout policy. Current generated PDFs use `treinamentos_brasilvida.report.v1` and `reportlab.a4.branded.v1`.
- Evidence: persistent metadata, blob reference, content hash, status, versioning relation, deletion metadata, and audit trail.

## Compatibility

The current implementation keeps existing URLs and response bodies compatible. The hardening is additive for clients: PDF responses and JSON attachment contracts now expose document policy metadata, while legacy consumers can continue reading the original fields.

# 77 - Frontend navigation dashboard fallback integrity

## Bloco executado

Hardening de integridade de roteamento e redirects da SPA para remover dashboard/home como escape implicito em login, perda de hash e falhas locais de rota.

## Contexto causal

A frente de assets/publicacao ja fechou o grafo frio de build. A frente atual trata o segundo vetor observado: retorno inadvertido para `#/dashboard`.

Antes desta mudanca:

- hash vazio no bootstrap era tratado como `#/dashboard`;
- `GET /login` em frontend oficial redirecionava para `#/dashboard`;
- erros de auth via API mandavam para `#/login` sem registrar rota de retorno;
- login SPA priorizava o `next` retornado pela API, que normalmente caia no landing backend, antes de qualquer contexto SPA local;
- shell/navegacao tratavam rota vazia como dashboard ativo.

## Mudanca aplicada

1. Estado de navegacao:
- criado `frontend/src/state/navigation-state.js`;
- guarda rota de retorno e ultima rota renderizada com sucesso em `sessionStorage`;
- nao renderiza UI, nao decide permissao e nao altera contrato de API.

2. Bootstrap SPA:
- remove fallback `routePath() || "#/dashboard"`;
- reconhece `/login` como entrada `#/login`;
- usuario nao autenticado vai para `#/login` preservando rota recuperavel;
- usuario autenticado em `#/login` volta para rota de retorno, ultima rota boa ou landing explicito;
- hash vazio autenticado usa ultima rota boa quando houver, e dashboard apenas como landing explicito sem contexto anterior;
- falha de rota/render continua renderizada em lugar, sem navegacao forcada.

3. Login/API:
- redirects de auth via API registram rota atual antes de ir para `#/login`;
- login SPA prioriza `next` do hash e rota de retorno antes do landing backend;
- destinos invalidos podem ser recusados sem cair automaticamente em dashboard.

4. Backend:
- `GET /login` com frontend oficial passa a redirecionar para `#/login`;
- `next` seguro e `auth_issue` permitido sao propagados dentro do hash do frontend.

## Prova material esperada

- `bootstrap.js` nao contem mais `routePath() || "#/dashboard"` nem `window.location.hash = "#/dashboard"`;
- `auth/routes.py` usa `redirect_to_frontend("#/login", query=_frontend_login_query())`;
- `api-client.js` chama `rememberCurrentRouteForLogin()` antes de redirecionar para login;
- `shell/login.js` usa `hashQuery().get("next")` e `consumeReturnRoute()`;
- testes de contrato validam que erro local/guard nao navega para dashboard.

## Regressao minima

- Nenhum endpoint JSON/API foi alterado.
- Permissoes de rota nao foram alteradas.
- Logout SPA continua usando `POST /api/v1/session/logout` e redirect final para `#/login`.
- `/dashboard` backend continua redirect legitimo para dashboard quando acessado explicitamente.

## Criterio de fechamento

`fechado` no escopo controlado se:

- login oficial entra em `#/login`, nao em `#/dashboard`;
- auth redirect preserva rota de retorno;
- rota vazia nao e mais tratada como fallback universal;
- erros de rota/render/guard nao executam navegacao para dashboard.

## Divida remanescente

- Redirects SSR legados ainda podem usar `dashboard.dashboard` como fallback operacional em fluxos backend fora da SPA. Eles ficam classificados como legado/SSR e devem ser tratados em frente propria para nao misturar semantica SPA com templates antigos.

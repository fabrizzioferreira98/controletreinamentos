# 18.5.1 Governanca de Dados

## 1. MAPA DE PAPEL DOS DADOS

- Usuarios e permissoes -> fonte de verdade de autenticacao/autorizacao -> tabela `usuarios`; `models.py` so usa snapshot de sessao como fallback curto.
- Tripulantes -> fonte de verdade cadastral sensivel -> tabela `tripulantes` com CPF, email, telefone, status, funcao, categoria e metadata de foto.
- Bases, pilotos e historico de status -> fonte de verdade operacional + historico de movimentacao -> tabelas `bases`, `pilotos` e `historico_status_piloto`.
- Equipamentos, tipos, segmentos e horas de voo -> fonte de verdade de catalogo/estrutura de treinamento -> tabelas `equipamentos`, `tipos_treinamento`, `segmentos_teoricos`, `horas_voo_aeronave`.
- Treinamentos -> fonte de verdade transacional -> tabela `treinamentos`; status de vencimento e visoes de dashboard sao derivados de datas e joins.
- Missoes, pernoites, adicionais, regras e parametros -> fonte de verdade da produtividade -> tabelas `missoes_operacionais`, `missao_tripulantes`, `pernoites_operacionais`, `produtividade_adicionais_excepcionais`, `produtividade_regras`, `produtividade_parametros`.
- Conferencias de produtividade -> historico/estado de validacao operacional -> tabela `produtividade_conferencias`, usada por relatorios e auditada em `application/relatorios.py`.
- Notificacoes -> fonte de verdade de destinatarios + historico minimo de envio -> tabelas `notificacoes_email` e `notificacoes_treinamento`; `sistema_controle` guarda `notification_last_*`.
- Auditoria -> trilha de auditoria, nao fonte de estado corrente -> tabela `auditoria_eventos`; `audit.py` normaliza e mascara campos sensiveis.
- Jobs -> estado operacional da fila + historico de execucao -> `background_jobs` e `background_job_executions`.
- Backups -> historico operacional de continuidade -> `backups_execucoes`; artefatos reais ficam fora do banco em dump/assets/config/manifest.
- Documentos e anexos persistidos -> fonte composta metadata + blob -> `tripulante_arquivos_pdf` e `treinamento_anexos_pdf` guardam metadata/hash/status/storage_ref; `MEDIA_STORAGE_ROOT` guarda blobs novos.
- Fotos de tripulante -> fonte composta metadata + blob -> `tripulantes.foto_storage_ref/foto_mime_type/possui_foto` + blob em filesystem; `foto_base64` e compat residual.
- PDFs gerados -> snapshot temporario de leitura, nao evidencia persistida -> `pdf_document_policy.py` classifica exports como `temporary_response` e `not_persisted`.
- Cache de dashboard/nav/painel -> cache, nao fonte de verdade -> `core/cache_service.py` guarda payload em memoria com TTL e invalidacao explicita.
- `sistema_controle` -> uso misto: cache legado/control state -> `cache.py` usa chaves `cache:*`; `infra/mailer.py` usa `notification_last_*`.

## 2. RETENCAO / HISTORICO / AUDITORIA

- Auditoria de mutacoes -> `auditoria_eventos` persiste anterior/novo mascarado -> lacuna: nao ha politica explicita de retencao/expurgo por idade.
- Historico de piloto -> `historico_status_piloto` preserva status/base anterior e nova -> lacuna: cobre movimentacao de piloto, nao todo historico cadastral.
- Conferencias de produtividade -> `produtividade_conferencias` mantem uma conferencia por tripulante/competencia -> lacuna: substituicao por `upsert` nao preserva versoes anteriores fora da auditoria.
- Documentos de tripulante -> replace marca anterior como `substituido` e retem blob -> lacuna: limpeza governada de substituidos/removidos ainda depende de decisao futura.
- Anexos de treinamento -> append-only/soft delete com retencao governada -> lacuna: remocao fisica so e clara em falha de escrita ou hard delete do treinamento pai.
- Fotos -> troca remove blob antigo apos commit -> lacuna: historico visual da foto nao e retido por desenho atual.
- Backups -> implementacao limpa artefatos por `BACKUP_RETENTION_DAYS`; runbook define backup diario, copia externa e drill semanal -> lacuna: retencao em tiers depende de operacao/config real.
- Jobs e erros de request -> `background_job_executions` e `request_error_events` preservam historico operacional -> lacuna: sem retencao temporal declarada.
- Dados sensiveis -> audit mascara CPF/email/telefone/foto/arquivo; file access usa `private, no-store` -> lacuna: snapshots/exportacoes fora do fluxo principal continuam exigindo controle operacional.

## 3. SNAPSHOTS / CACHE / COMPAT

- Backup dump/assets/config/manifest -> snapshot legitimo de restore -> risco medio: precisa consistencia temporal entre banco e uploads.
- Drill de restore -> snapshot validado operacionalmente -> risco medio: restore completo so fecha quando `--restore-url` e usado; fallback `.json.gz` fica parcial.
- PDF exportado -> snapshot legitimo temporario -> risco baixo: nao e persistido nem evidencia formal.
- Auth session snapshot -> snapshot legitimo de disponibilidade curta -> risco medio: pode manter permissoes antigas ate `AUTH_SNAPSHOT_MAX_AGE_SECONDS`.
- Dashboard/painel/nav -> cache em memoria -> risco baixo/medio: dado pode ficar stale ate TTL ou invalidacao.
- `sistema_controle cache:*` -> cache persistido legado -> risco medio: mesma tabela tambem guarda estado operacional de notificacao.
- `sync_tripulantes_snapshot.py` -> snapshot perigoso -> risco alto: materializa PII, foto/base64/storage_ref e pode aplicar upsert em destino.
- `cleanup_operational_data.py` -> cleanup perigoso com snapshot minimo de admin/contagens -> risco alto: apaga dados operacionais amplos mediante confirmacao.
- `foto_base64` -> compat residual -> risco alto: dado sensivel duplicado/legado ainda lido por fluxos de foto.
- `arquivo_pdf` com `storage_ref = 'db:bytea'` -> compat residual -> risco alto: blob legado compete com storage novo enquanto existir.
- `fs:` legado -> compat residual controlada -> risco medio: leitura continua se estiver sob a raiz operacional.
- `s3://`/`remote:` documental -> compat/reserva nao ativa -> risco medio: metadata reconhecida, mas blob remoto nao e servido sem adapter.

## 4. BANCO vs STORAGE

- Documentos PDF -> consistencia: banco e fonte da metadata/hash/status/storage_ref; filesystem e fonte do blob novo -> impacto: documento so e valido quando metadata e blob convergem.
- `db:bytea` legado -> divergencia controlada: banco ainda pode ser fonte do blob quando `storage_ref = 'db:bytea'` -> impacto: restore/performance/migracao continuam mais caros.
- Fotos -> divergencia residual: `foto_storage_ref` novo convive com `foto_base64` antigo -> impacto: risco de duplicidade, referencia quebrada ou leitura defensiva.
- Orfaos de storage -> divergencia detectavel: `document_blobs.py` classifica `metadata_without_blob`, `metadata_without_reference`, `orphan_blobs` -> impacto: nao deve servir blob sem metadata confiavel.
- Backup -> consistencia operacional: banco, assets e config sao artefatos separados no mesmo job -> impacto: restore incompleto se dump e uploads nao forem restaurados como conjunto.
- Storage remoto documental -> divergencia assumida: refs remotas sao metadata nao verificadas -> impacto: nao ha disponibilidade documental ate existir adapter.

## 5. POLITICA DE RESTORE

- Estado atual -> `run_backup_job` gera dump, assets, config e manifest; registra `backups_execucoes`; limpa por `BACKUP_RETENTION_DAYS`; pode espelhar externo/S3 -> risco: sucesso local nao garante restore se assets/config ficarem fora do escopo real -> lacuna: validar configuracao real por ambiente.
- Estado atual -> runbook Windows documenta restore manual de banco, arquivos/config e drill semanal -> risco: procedimento depende de execucao disciplinada, nao apenas codigo -> lacuna: evidencia de drill recente fica fora desta etapa.
- Estado atual -> `backup_restore_drill.py` valida `pg_restore --list`, extrai arquivos e pode restaurar em banco temporario -> risco: sem `--restore-url`, restore completo fica marcado como parcial -> lacuna: restore real precisa ambiente isolado.
- Estado atual -> fallback logico JSON/SQLite existe para contingencia -> risco: drill automatizado nao restaura completamente `.gz` -> lacuna: fallback nao e politica primaria de restore.
- Estado atual -> consistencia documental e monitorada por inventario metadata/blob -> risco: backup antigo pode restaurar metadata que aponta para blob ausente se assets nao acompanharem -> lacuna: restore deve validar documentos criticos apos subir.

## 6. CONCLUSAO DA ETAPA 18.5.1

- Governanca clara -> fontes de verdade principais estao nas tabelas de dominio; cache de dashboard/nav/painel nao e fonte; PDFs gerados sao snapshots temporarios; documentos persistidos exigem par metadata/blob; backups sao snapshots legitimos de continuidade.
- Governanca perigosa -> compat de `foto_base64` e `db:bytea`, snapshots de sincronizacao de tripulantes, cleanup amplo, uso misto de `sistema_controle`, ausencia de retencao explicita para auditoria/jobs/erros e restore parcial sem drill completo continuam como risco conhecido.

A 18.5.1 fica concluida como classificacao: cada grupo relevante foi separado entre fonte de verdade, cache, snapshot legitimo, snapshot perigoso e compat residual, com lacunas de retencao, historico, auditoria, banco/storage e restore formalizadas para tratamento futuro.

# 102 - Consistencia de save do status operacional de tripulantes

## Contexto

Relato operacional: ao editar um tripulante de `Treinamento` para `Ativo`, marcar `elegivel_adicional_excepcional` e salvar, a listagem continuava exibindo `Treinamento`.

## Causa

O frontend enviava `status` corretamente para `PUT /api/v1/tripulantes/<id>`. O backend validava o campo, mas `sync_linked_pilot_from_tripulante` preservava `pilotos.status` existente quando o tripulante estava ativo. Como o contrato de leitura define `pilotos.status` como owner canonico, a listagem lia o status antigo.

## Correcao

`sync_linked_pilot_from_tripulante` passa a aplicar primeiro o status submetido no formulario para tripulantes ativos. A inativacao continua forçando `afastado`.

## Migration de schema

Nao houve alteracao de schema. O owner canonico ja era `pilotos.status`, e o snapshot residual `tripulantes.status` continua sendo atualizado por compatibilidade.

## Regressao

Guardas focais:

- `tests/unit/test_tripulante_operational_status_owner.py::test_sync_linked_pilot_from_tripulante_applies_submitted_status_and_preserves_base_owner`
- `tests/unit/test_tripulante_operational_status_owner.py::test_sync_linked_pilot_from_tripulante_regression_training_to_active_with_exceptional_flag`

## Criterio de fechamento

Editar um tripulante de `Treinamento` para `Ativo` e salvar deve fazer a resposta de detalhe e a listagem retornarem `status = Ativo`, com `status_operacional_owner = pilotos.status`.

# 31.0 - Abertura formal da Frente 31

## 1. Objetivo central

Abrir formalmente a Frente 31 como frente documental de governanca de compatibilidade, legado vivo, archive/topologia, excecoes locais controladas e backlog de remocao futura.

A Frente 31 nao e release operacional de produto. Ela existe para reduzir ambiguidade sobre residuos ja existentes no repo, sem reabrir frentes encerradas e sem mudar comportamento funcional.

Delta concreto usado como base:

- `docs/governance/removal-backlog.md` ja lista candidatos a remover, isolar, congelar ou morrer depois, mas nao autoriza limpeza fisica automatica.
- `scripts/README.md` classifica `scripts/` como compatibilidade controlada e lista wrappers ainda protegidos por risco de consumidor externo.
- `legacy/LIVE_LEGACY.md` registra legado vivo real: SSR, operacoes SSR compat, aba File legada e compat de midia/status.
- `docs/governance/repo-topology.md` separa topologia oficial, compat, legacy, archive, excecoes locais `.venv/` e `frontend/dist/`.
- `archive/` e `docs/migration/retired-platforms/` permanecem historico classificado, sem autoridade operacional viva.

## 2. Escopo principal

| area | escopo da Frente 31 |
| --- | --- |
| Compatibilidade | Classificar compat residual ja documentado, principalmente wrappers em `scripts/`, sem remover nesta abertura. |
| Legacy vivo | Ratificar que `legacy/` existe por fluxos vivos documentados e nao e backlog generico. |
| Archive/topologia | Separar historico classificado de fonte viva e impedir segunda raiz operacional. |
| Excecoes locais | Manter `.venv/` e `frontend/dist/` como excecoes locais/geradas, nao como topologia oficial. |
| Backlog de remocao futura | Tratar o backlog como fila condicionada por pre-condicao, evidencia e risco, nunca como permissao de apagar. |
| Governanca documental | Registrar criterio de aceite, gates continuos e risco permitido para etapas 31.x. |

## 4. Fora de escopo explicito

| fora de escopo | motivo |
| --- | --- |
| GO operacional de release, deploy, CD ou validacao de produto | A Frente 31 e governanca/compat/topologia, nao release operacional. |
| Uso de `run_release_strict.py` como regua central | Esse script pertence a release operacional e so se aplica se uma etapa tocar release. |
| Reabertura das Frentes 18, 27, 28, 29 ou 30 | A Frente 31 usa decisoes anteriores como base documental fechada. |
| Alteracao de codigo de produto ou contrato funcional | Esta etapa e abertura formal documental. |
| Alteracao de wrappers, archive ou legacy | Esta etapa nao remove, move, reclassifica fisicamente nem edita esses artefatos. |
| Refatoracao, performance, acessibilidade, schema, storage ou migracao de dados | Pertencem a frentes proprias quando houver escopo material. |
| Limpeza em lote | O backlog exige item, pre-condicao, evidencia e risco antes de qualquer remocao futura. |

| README de migracao indexa a abertura | Conferir entrada de `31.0-abertura-formal-frente-31.md` em `docs/migration/README.md`. |

## 6. Gates continuos

| gate | regra para a Frente 31 |
| --- | --- |
| Gate documental local | Toda etapa 31.x deve atualizar o documento de migracao correspondente e indice aplicavel. |
| Gate de fonte viva | Mudanca documental nao pode promover `scripts/`, `legacy/`, `archive`, `.venv/` ou `frontend/dist/` a trilha canonica. |
| Gate de pre-condicao | Remocao futura so pode ser discutida com item exato, consumidor auditado, evidencia e risco registrado. |
| Gate de compatibilidade | Wrapper de compat permanece bloqueado quando consumidor externo nao estiver comprovadamente ausente ou migrado. |
| Gate de legacy vivo | Fluxo registrado em `legacy/LIVE_LEGACY.md` nao pode ser removido por classificacao estetica. |
| Gate de archive/topologia | Archive continua historico classificado e nao vira segunda raiz viva. |
| Gate de release | `run_release_strict.py` nao e exigencia geral desta frente; so aplica se uma etapa futura tocar release operacional. |

## 7. Risco de regressao

### Aceitavel

- Ajuste textual em `docs/migration` sem mudar fonte viva, comando, teste, wrapper, legacy ou archive.
- Atualizacao de indice documental que apenas aponta a abertura correta.
- Registro de bloqueio por falta de pre-condicao, sem declarar remocao.

### Tolerado com compensacao

- Consumidor externo de wrapper nao auditado: manter wrapper bloqueado e registrar a incerteza como impedimento de remocao.
- Archive com valor historico incerto: manter congelado e nao tratar como fonte viva.
- Excecao local existente, como `.venv/` ou `frontend/dist/`: manter classificada como local/gerada e fora da topologia oficial.
- Gate operacional fora do escopo da 31: registrar como nao aplicavel, sem converter falha de release em decisao desta frente.

### Proibido

- Declarar GO operacional de release ou deploy a partir desta abertura.
- Usar `run_release_strict.py` como criterio central da Frente 31.
- Reabrir Frentes 18, 27, 28, 29 ou 30.
- Remover, mover ou editar wrappers, legacy ou archive nesta etapa.
- Alterar codigo de produto, contrato funcional, schema, storage, API, frontend, jobs, backup ou restore.
- Tratar backlog de remocao como autorizacao de limpeza fisica sem pre-condicao e evidencia.

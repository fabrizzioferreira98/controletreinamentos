# 59 - Frontend asset graph closed publish contract

## Bloco executado

Hardening complementar do contrato de build/publicacao do frontend SPA para fechar a brecha de mistura entre assets antigos e novos no first load frio.

## Contexto causal

A auditoria forense global apontou que telas quebradas no primeiro acesso estavam materialmente ligadas a grafo de assets incoerente:

- `index.html` podia chegar com uma combinacao diferente de CSS/JS do diretorio ativo;
- builds diferentes alteravam `app.css`, enquanto varios JS mantinham o mesmo digest de conteudo;
- navegacao interna era menos afetada porque os modulos lazy ja carregados ficavam em cache runtime;
- o contrato anterior ja tinha fingerprint e swap atomico, mas a validacao de publicacao ainda aceitava JS/CSS orfaos no diretorio de build informado por `-SkipBuild -BuildDir`.

## Mudanca aplicada

1. Publicacao:
- `ops/windows/scripts/Publish-Frontend.ps1` agora rejeita qualquer arquivo `.js` ou `.css` que exista no build/staging/prod e nao esteja listado em `asset-manifest.json`;
- `index.html` agora e validado como grafo fechado: ele so pode referenciar os entrypoints `app.js`, `app.css` e `config.js` declarados no manifest;
- a validacao roda no build, no staging e no destino ativo publicado.

2. Borda/cache:
- `ops/windows/caddy/Caddyfile.example` deixou de servir caminhos frontend-owned legados nao fingerprintados como `/app.js`, `/app.css`, `/config.js` e modulos soltos;
- requests GET/HEAD para `.js`/`.css` frontend-owned nao fingerprintados passam a receber resposta explicita `410` com `no-store`, depois da regra dos assets fingerprintados imutaveis;
- `asset-manifest.json` e `login-citation-jet.jpg` seguem como assets mutaveis revalidaveis.

3. Testes:
- o build agora tem teste de grafo fechado, comparando todos os `.js`/`.css` em disco contra `fingerprinted_assets`;
- o Caddy agora tem teste impedindo reintroducao de serving legado de JS/CSS nao fingerprintado;
- o publish script agora tem teste exigindo as validacoes anti-orfaos e de entry HTML fechado.

## Prova material esperada

Um build valido deve satisfazer:

- `index.html` sem `?v=`;
- `index.html` referenciando somente entrypoints fingerprintados do manifest;
- nenhum `.js` ou `.css` em disco fora de `asset-manifest.json`;
- entry HTML com `Cache-Control: no-store, no-cache, must-revalidate, max-age=0`;
- assets fingerprintados com `Cache-Control: public, max-age=31536000, immutable`;
- publicacao por staging + swap atomico do diretorio `prod`.

## Regressao minima

- Nenhum contrato HTTP da API foi alterado.
- Nenhuma tela ou UX especifica foi alterada.
- A mudanca fica restrita a build, publicacao, Caddy e testes de integridade de assets.

## Criterio de fechamento

No escopo local controlado, o fechamento exige prova de cold load via HTTP:

- GET `/` retorna `index.html` revalidavel/no-store;
- os assets referenciados no primeiro HTML retornam 200;
- reload de `/index.html` aponta para o mesmo grafo;
- uma nova publicacao simulada nao deixa `.js`/`.css` orfaos no destino ativo.

## Divida remanescente

- A producao live ainda precisa ser sondada apos aplicacao do Caddy real, porque a sessao anterior nao conseguiu validar o dominio publico.
- Se algum proxy externo ignorar `no-store`, o contrato correto passa a depender da governanca desse edge.

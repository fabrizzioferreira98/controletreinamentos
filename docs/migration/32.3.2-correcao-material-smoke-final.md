## 2. Mudanca aplicada

### 2.1 Motivo material do `403`

O `403` em `/api/internal/metrics` foi causado por ausencia de token valido no processo do smoke rerodado na etapa `32.2.4`.

Provas:

| prova | leitura |
| --- | --- |
| Codigo do endpoint `/api/internal/metrics` | O endpoint compara `X-Metrics-Token` com `METRICS_TOKEN`; token ausente ou divergente retorna `403 metrics_forbidden`. |
| `metrics_hardening.json` historico | Confirma que o runtime homolog tinha `METRICS_TOKEN` configurado e que token valido retorna `200`. |
| `post_release_smoke_32_2_4.json` | Registrou `internal_metrics` com status `403`, `prometheus_content=false` e `runtime_resource_metric_present=false`. |
| `C:\srv\controle-treinamentos\env\hml.env` | Env vivo do servico HML contem `METRICS_TOKEN` configurado. |
| `CT-App-Hml.xml` | Servico `CT-App-Hml` usa `C:\srv\controle-treinamentos\env\hml.env` como `EnvFile`. |

### 2.2 Correcao minima

Foi aplicada apenas a correcao minima no caminho de smoke/configuracao:

- carregar o `METRICS_TOKEN` real a partir de `C:\srv\controle-treinamentos\env\hml.env`;
- injetar o token somente no ambiente do processo do `post_deploy_smoke.py`;
- manter o mesmo `release_id`, mesmo ambiente, mesmo manifest e mesmo runtime esperado;
- nao registrar o valor do token em documento, comando persistente ou artefato.

Nenhum codigo de produto foi alterado.

# Frente 18.6.P0.4 - Separacao da trilha de migrations

## 1. CLASSIFICACAO DOS ARTEFATOS

- `backend/tools/maintenance/bootstrap_db_schema.py` -> bootstrap estrutural -> comando canonico novo para subir apenas schema e indices previsiveis.
- `backend/src/controle_treinamentos/db/schema_bootstrap.py` -> bootstrap estrutural -> implementacao explicita da trilha estrutural segura.
- `backend/src/controle_treinamentos/db/migrations.py` -> migracoes corretivas -> mantido como trilha manual para alteracoes corretivas e compat de bancos historicos, sem seed historico embutido.
- `backend/src/controle_treinamentos/db/seeder.py` -> seeds operacionais controlados -> bootstrap de admin/defaults/bases; nao define evolucao estrutural principal.
- `backend/src/controle_treinamentos/db/training_program_seed.py` -> seed historico -> referencia operacional herdada, fora do bootstrap estrutural.
- `ops/scripts/database/sync_training_master_types.py` -> sync operacional -> altera catalogo mestre; nao e migracao estrutural.
- `ops/scripts/database/import_tripulantes_csv.py` -> seed/import historico -> carga massiva de dado, fora da trilha estrutural.
- `ops/scripts/database/cleanup_operational_data.py` -> reparo/manual perigoso -> limpeza ampla de dado operacional, explicitamente fora da trilha principal.
- `ops/scripts/database/sync_tripulantes_snapshot.py` -> sync operacional perigoso -> snapshot compat/legado sem papel na evolucao estrutural.
- `ops/scripts/database/migrate_tripulante_media_to_storage.py` -> migracao de compat -> migracao pontual de legado vivo.
- `ops/scripts/database/reconcile_tripulante_photos.py` -> compat/reconciliacao -> reparo manual de legado.
- `ops/scripts/database/run_db_consistency.py --repair` -> reparo manual -> executa bootstrap estrutural + migracoes corretivas antes da validacao, sem rodar seed historico.

## 2. TRILHA PRINCIPAL AJUSTADA

- Continua como bootstrap estrutural:
  - `backend/tools/maintenance/bootstrap_db_schema.py`
  - `backend/src/controle_treinamentos/db/schema_bootstrap.py`
- Saiu da trilha principal:
  - `training_program_seed.py`
  - `sync_training_master_types.py`
  - `import_tripulantes_csv.py`
  - `cleanup_operational_data.py`
  - `sync_tripulantes_snapshot.py`
  - `run_db_consistency.py --repair` como caminho padrao
- Motivo:
  - bootstrap estrutural agora sobe apenas schema e indices canonicamente previsiveis;
  - seed, sync, import e reparo amplo ficam tratados como trilhas manuais ou historicas.

## 3. ITENS CONGELADOS

- `backend/src/controle_treinamentos/db/training_program_seed.py` -> seed historico -> nao volta ao bootstrap estrutural; sobrevive apenas ate substituicao controlada.
- `ops/scripts/database/sync_training_master_types.py` -> sync operacional -> nao reincorpora ao bootstrap nem ganha novo uso automatico.
- `ops/scripts/database/import_tripulantes_csv.py` -> importacao massiva -> continua fora da trilha principal e sem promocao para fluxo padrao.
- `ops/scripts/database/cleanup_operational_data.py` -> limpeza destrutiva -> uso manual e excepcional, sem expansao.
- `ops/scripts/database/sync_tripulantes_snapshot.py` -> snapshot perigoso -> segue fora da trilha principal e em trilha de aposentadoria.
- `ops/scripts/database/migrate_tripulante_media_to_storage.py` e `ops/scripts/database/reconcile_tripulante_photos.py` -> compat historica -> nao viram rotina recorrente.

## 4. REDUCAO DE RISCO DE EXECUCAO

- Antes -> `execute_migrations` misturava bootstrap estrutural com corretivas e `training_program_seed`.
- Depois -> `execute_schema_bootstrap` sobe schema/indices; `execute_migrations` fica corretivo; `execute_seed_bootstrap` fica seed operacional; `run_db_consistency --repair` fica reparo manual sem seed historico.
- Ganho operacional:
  - operador nao confunde bootstrap com seed/import/sync;
  - `migrations.py` deixa de puxar referencia historica de programa;
  - `run_db_consistency --repair` para de carregar seed herdado como se fosse parte natural de reparo estrutural;
  - documentacao passa a indicar um comando proprio para bootstrap estrutural.

## 5. DOCUMENTACAO OPERACIONAL

- `docs/operations/canonical-commands.md` -> adiciona `bootstrap_db_schema.py` e rebaixa `--repair` a reparo manual -> separa trilha principal de excecao.
- `docs/operations/RUNBOOK.md` -> explicita diferenca entre bootstrap estrutural, consistencia e reparo -> reduz ambiguidade operacional.
- `docs/operations/WINDOWS_SELF_HOSTED_SERVER.md` -> separa exemplo de consistency check, bootstrap estrutural e repair manual -> evita uso de `--repair` como default.
- `ops/README.md` -> redefine `scripts/database/` como area manual, nao bootstrap estrutural -> impede mistura conceitual.
- `README.md` -> passa a listar bootstrap estrutural como tarefa propria e separa bootstrap de repair -> reforca a trilha canonica.

## 6. MUDANCAS APLICADAS

- `backend/src/controle_treinamentos/db/schema_bootstrap.py` -> novo modulo de bootstrap estrutural e validacao de schema.
- `backend/src/controle_treinamentos/db/migrations.py` -> seed historico removido e papel rebaixado para migracoes corretivas.
- `backend/src/controle_treinamentos/db/seeder.py` -> bootstrap estrutural separado do seed operacional.
- `backend/src/controle_treinamentos/db/__init__.py` -> exports alinhados com a nova trilha canonica.
- `backend/tools/maintenance/bootstrap_db_schema.py` -> novo comando oficial para bootstrap estrutural.
- `ops/scripts/database/run_db_consistency.py` -> reparo passa a usar bootstrap estrutural + corretivas, sem seed operacional.
- `docs/operations/canonical-commands.md`, `docs/operations/RUNBOOK.md`, `docs/operations/WINDOWS_SELF_HOSTED_SERVER.md`, `ops/README.md`, `README.md` -> docs alinhadas ao novo desenho.
- `tests/unit/test_db_migrations_schema_split.py` -> cobertura da separacao entre bootstrap, repair e seed historico.

## 7. VALIDACAO

- Cenario -> `migrations.py` como saco misto com seed historico -> resultado: removido o uso de `training_program_seed` -> evidencia: o modulo nao importa nem chama mais `seed_training_program_reference`.
- Cenario -> bootstrap estrutural padrao -> resultado: ganhou comando dedicado `bootstrap_db_schema.py` -> evidencia: docs operacionais e teste apontam a nova entrada canonica.
- Cenario -> reparo manual -> resultado: `run_db_consistency --repair` usa `repair_and_validate_schema` sem seed operacional -> evidencia: o script nao chama mais `execute_script()`.
- Cenario -> seed operacional -> resultado: `seeder.py` sobe schema pela trilha estrutural e mantem seeds fora de `migrations.py` -> evidencia: `execute_seed_bootstrap` chama `execute_schema_bootstrap`.

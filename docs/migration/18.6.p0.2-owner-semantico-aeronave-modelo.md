# 18.6.P0.2 - Owner Semantico de `treinamentos.aeronave_modelo`

Base fixa: Frente 18 encerrada como diagnostico. Esta etapa nao reabre a analise estrutural. Ela fecha o papel do campo no modelo atual.

## 1. DECISAO SEMANTICA

- `treinamentos.aeronave_modelo` permanece no modelo atual como `snapshot` do modelo de aeronave usado no treinamento realizado
- owner canonico do dominio continua fora de `treinamentos`: `horas_voo_aeronave` por `tipo_treinamento_id + aeronave_modelo`
- o nome fisico no banco ainda nao muda agora; a semantica de aplicacao e contrato passa a tratar o campo como `aeronave_modelo_snapshot`

## 2. ALINHAMENTO COM O PROGRAMA

- `tipos_treinamento.exige_equipamento` continua decidindo se o fluxo precisa selecionar um modelo canonico de referencia
- `horas_voo_aeronave.aeronave_modelo` continua como referencia canonica do programa e alimenta template, horas e CTAC
- `segmentos_teoricos` continuam como estrutura canonica do treinamento por programa; nao sao derivados do snapshot
- CTAC continua dependente da referencia de `horas_voo_aeronave`, nao do snapshot como owner
- o efeito da decisao e este: o programa usa `aeronave_modelo_referencia`; o registro persistido guarda `aeronave_modelo_snapshot`

## 3. LEITURAS REVISADAS

- template de programa -> leitura de referencia -> usar `aeronave_modelo_referencia` como entrada explicita
- listagem/detalhe de registros por programa -> leitura legitima de snapshot -> expor `aeronave_modelo_snapshot` e manter `aeronave_modelo` apenas como alias de compat
- opcoes e cadastro mestre de horas -> leitura canonica -> expor `aeronave_modelo_referencia`
- API generica de treinamentos -> leitura explicita -> expor `origem_registro` e, quando houver programa, expor `aeronave_modelo_snapshot`

## 4. ESCRITAS REVISADAS

- batch e update de `treinamentos-tripulantes` -> entrada de referencia -> persistir no banco como snapshot
- fluxo generico de `treinamentos` -> nao pode mais escrever `aeronave_modelo`, `segmento_teorico_id` ou `ctac_*`
- update generico em registro de programa -> bloqueado; o owner semantico desse fluxo fica em `treinamentos-tripulantes`

## 5. TRILHA FUTURA

- o campo permanece na trilha principal apenas como snapshot explicito
- rename futuro esperado: `aeronave_modelo_snapshot`
- condicao de saida da trilha principal como coluna relevante de leitura: quando o treinamento realizado puder apontar para uma estrutura canonica suficiente sem depender do texto persistido para enriquecimento operacional

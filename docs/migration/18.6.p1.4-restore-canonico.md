# 18.6.P1.4 - Formalizar restore canonico

## Definicao de restore canonico

- componente `dump` -> reconstrui metadata relacional -> obrigatorio
- componente `assets` -> reconstrui blobs privados e documentais -> obrigatorio
- componente `config` -> reconstrui ambiente operacional restauravel -> obrigatorio
- componente `manifest` -> fixa a janela e a composicao do pacote -> obrigatorio
- componente `post_restore_validation` -> prova consistencia banco/storage apos restore -> obrigatorio

Classificacao operacional:
- dump isolado -> restore parcial
- assets isolados -> restore auxiliar
- pacote completo sem pos-validacao -> restore auxiliar composto
- pacote completo com pos-validacao sem inconsistencia critica -> restore canonico

## Validacao pos-restore

- fotos -> `foto_storage_ref` precisa resolver blob; `foto_base64` aparece apenas como compat residual visivel
- documentos/anexos -> `storage_ref` precisa resolver blob; `db:bytea` aparece apenas como compat residual visivel
- `metadata_without_blob` -> falha critica
- `metadata_without_reference` -> falha critica
- `metadata_with_unsupported_reference` -> falha critica
- `orphan_blobs` -> aviso operacional e reconciliacao manual
- artefato fora da janela do manifesto -> restore invalido

## Banco vs storage

- banco e storage ficam acoplados por regra: ponteiro em banco sem blob correspondente nao conta como restore concluido
- owner canonico continua separado:
  - metadata em banco
  - blob privado/documental em storage
  - manifesto como amarra da janela restauravel
- restore de continuidade deixa de ser "subir dump e torcer"

## Passo operacional fechado

- `backup_restore_drill.py` agora valida o pacote minimo de restore canonico antes de seguir
- `run_restore_postcheck.py` vira a validacao objetiva de metadata/blob no ambiente restaurado
- `windows_backup_restore_rollback.md` passa a descrever pre-condicoes, ordem, checagens, sinais de restore incompleto e criterio de sucesso

## Divida deliberadamente adiada

- automatizar restore completo ponta a ponta no mesmo script
- validar refs remotas somente quando existir adapter operacional real
- acoplar o post-check diretamente ao drill com restore real em todos os ambientes

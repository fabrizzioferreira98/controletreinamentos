# 54 - Frontend asset fingerprint + atomic publish + edge cache split

## Bloco executado

Hardening estrutural de integridade de build/publicacao do frontend separado, com foco em first load frio coerente.

## Contrato anterior

- assets criticos da SPA dependiam de `?v=` em HTML/JS;
- publicacao por espelhamento direto no diretorio ativo podia expor janela de troca nao atomica;
- cache de borda tinha `no-store` na entrada HTML, mas sem contrato forte para assets fingerprintados imutaveis.

## Mudanca aplicada

1. Build/frontend:
- `frontend/scripts/build_frontend.py` passou a gerar JS/CSS apenas com fingerprint no nome (`<arquivo>.<build_utc>.<hash>.ext`);
- `index.html` passou a apontar somente para entrypoints fingerprintados;
- `asset-manifest.json` passou a ser artefato obrigatorio com `entrypoints` e `fingerprinted_assets`;
- `frontend/src` deixou de usar `?v=` para versionamento principal dos assets da SPA.

2. Publicacao/frontend:
- `ops/windows/scripts/Publish-Frontend.ps1` passou a validar manifesto (`frontend_asset_manifest_v1`) e a rejeitar build com `?v=`;
- publicacao passou a usar staging + swap atomico do diretorio ativo (`prod`);
- release anterior passa a ser preservado com nome versionado para rollback e backup.

3. Borda/cache:
- `ops/windows/caddy/Caddyfile.example` manteve `entry HTML` em `no-store`;
- assets fingerprintados passaram para `Cache-Control: public, max-age=31536000, immutable`;
- assets nao fingerprintados passaram para cache revalidavel (`no-cache, max-age=0, must-revalidate`).

## Prova material

- `frontend/scripts/build_frontend.py` contem pipeline com:
  - `build_asset_payloads(env)`;
  - `build_fingerprint_map(asset_payloads, build_version)`;
  - `write_fingerprinted_assets(...)`;
  - `rewrite_html_references(...)`;
  - `asset-manifest.json`.
- `ops/windows/scripts/Publish-Frontend.ps1` contem:
  - validacao de schema `frontend_asset_manifest_v1`;
  - bloqueio de `?v=` no artefato;
  - swap atomico por `Rename-Item` entre `staging` e destino ativo.
- `ops/windows/caddy/Caddyfile.example` contem:
  - matcher `@frontend_assets_immutable` com `path_regexp frontend_immutable`;
  - header `Cache-Control "public, max-age=31536000, immutable"`;
  - matcher `@frontend_assets_mutable` com cache revalidavel.
- `tests/contract/test_frontend_build_runtime_env_boundaries.py` cobre:
  - entrypoints fingerprintados no build;
  - ausencia de `?v=` no output;
  - contrato de cache no Caddy;
  - contrato de swap atomico no publish script.

## Regressao minima

- nenhum endpoint backend/API foi alterado;
- nenhum contrato HTTP de negocio foi alterado;
- mudanca restrita a build/publish/cache de assets frontend.

## Criterio de fechamento

Status no escopo de codigo: `fechado`.
Status no escopo de runtime frio real (borda/publicacao real): `pendente_validacao_operacional`.

O contrato final passou a exigir:

- entry HTML revalidavel/no-store;
- assets criticos com identidade forte por nome fingerprintado;
- publicacao atomica sem mistura de grafo entre releases.

## Divida remanescente

- referencias `/static/*` compartilhadas com backend seguem como acoplamento residual controlado;
- evolucao para fingerprint de todos os assets estaticos nao JS/CSS depende de frente propria de ownership entre frontend e backend/static.

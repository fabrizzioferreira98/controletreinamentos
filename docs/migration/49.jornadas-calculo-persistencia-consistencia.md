# 49 - Jornadas: consistencia entre lancamento, calculo e exibicao

## Bloco executado

Auditoria e correcao focal da logica da aba `Lancamento de Jornada`, sem alterar contrato, rota, loader, permissao ou schema.

## Regra confirmada

- O calculo exibido na aba usa o motor backend de jornada via `build_produtividade_jornadas`.
- A tela envia campos opcionais de planilha como strings vazias quando o usuario limpa o input.
- Campos opcionais limpos devem persistir como `NULL`/vazio funcional, nao como valor anterior.
- `codigo_voo` pode existir como identificador legado da missao, mas nao deve repopular `N DB` quando `numero_db` foi explicitamente limpo.

## Problema corrigido

O backend tratava alguns campos vazios como ausentes e reaproveitava valores atuais em update. Na montagem da linha calculada, `numero_db` tambem podia cair para `codigo_voo`, fazendo o valor antigo reaparecer na tela e no relatorio.

## Mudanca aplicada

- `_lancamento_data` agora diferencia campo ausente de campo enviado vazio para `numero_db`, `relatorio_voo`, `adicional_pos_manual` e justificativa.
- `_build_line` passa a usar `codigo_voo` como fallback de `numero_db` apenas quando nao ha coluna/campo explicito de DB no payload material.
- A tela deixou de usar `codigo_voo` como fallback visual direto do input `N DB`.

## Prova material

- Teste backend cobre limpeza explicita de campos opcionais em update.
- Teste backend cobre `numero_db` vazio sem repopular por `codigo_voo`.
- Teste frontend cobre ausencia do fallback visual `item.numero_db || item.codigo_voo`.

## Migration de banco

Nao aplicavel. Nao houve DDL, DML estrutural, contrato novo ou alteracao de schema.

## Criterio de fechamento

Fechado para a inconsistencia corrigida neste bloco quando os testes focais de backend, frontend e contratos de PDF passam.

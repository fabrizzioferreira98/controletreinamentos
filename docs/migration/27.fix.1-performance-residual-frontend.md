# 27.FIX.1 - Performance residual de frontend

## 1. HOTSPOTS RESIDUAIS

Escopo fechado nesta frente:

- startup canonicamente pequeno: `app.js`, `lib.js` e `shell.js`;
- lazy import por rota para `pages-dashboard-tripulantes.js` e `pages-treinamentos-relatorios.js`;
- sessao redundante no login;
- waterfall residual de dashboard/detalhe;
- medicao por fase no proprio frontend.

Baseline capturado antes do ajuste em `frontend/dist`:

| fase | arquivos eager | bytes |
| --- | --- | ---: |
| startup antigo | `app.js`, `lib.js`, `shell.js`, `pages-dashboard-tripulantes.js`, `pages-treinamentos-relatorios.js` | `288482` |
| pagina dashboard/tripulantes | import estatico, carregado antes da rota | `82125` |
| pagina treinamentos/relatorios | import estatico, carregado antes da rota | `156372` |

## 2. MEDICAO POR FASE

O frontend agora publica `window.__FRONTEND_PERF__` a cada boot com estas fases:

- `startup`: tempo total do ciclo de boot atual;
- `session`: validacao de sessao;
- `route_resolve`: resolucao de hash route e permissao;
- `route_import`: import dinamico do modulo de rota, com `cache=miss` ou `cache=hit`;
- `route_render`: renderizacao da rota ja resolvida.

O script repetivel `frontend/scripts/measure_frontend_perf.py` mede o artefato final e valida a forma do caminho quente. Resultado apos build:

| fase | arquivos eager | bytes |
| --- | --- | ---: |
| startup canonico | `app.js`, `lib.js`, `shell.js` | `52819` |
| modulo lazy `tripulantes` | `pages-dashboard-tripulantes.js` | `82663` |
| modulo lazy `treinamentos` | `pages-treinamentos-relatorios.js` | `156594` |

## 3. AJUSTES APLICADOS

- `frontend/src/app.js` removeu imports estaticos de paginas e passou a carregar cada familia por `import()` no primeiro acesso da rota.
- `frontend/src/lib.js` ganhou medicao de fase, reset por boot e export em `window.__FRONTEND_PERF__`.
- `frontend/src/shell.js` deixou de fazer `fetch` proprio de sessao; o login reutiliza `refreshSession()` e so forca nova sessao no retry de CSRF.
- `frontend/src/pages-dashboard-tripulantes.js` paralelizou detalhe do tripulante, arquivos e opcoes padrao; a consulta especifica por base fica apenas como fallback de compat quando a base existente nao vier nas opcoes padrao.
- `frontend/src/pages-treinamentos-relatorios.js` paralelizou os detalhes em edicao da raiz de treinamentos com options/listas.

## 4. GANHO REGISTRADO

| item | antes | depois | ganho |
| --- | ---: | ---: | ---: |
| JS eager de startup | `288482` bytes | `52819` bytes | `-235663` bytes (`-81.7%`) |
| GET normal de sessao no login frio ate submeter | ate `3` pontos de chamada | `1` chamada de rede normal, com cache de CSRF | remove redundancia normal |
| detalhe de tripulante | detalhe -> opcoes/arquivos | detalhe + opcoes + arquivos em paralelo | uma fase a menos no caminho comum |
| raiz de treinamentos em edicao | listas -> tipo -> segmento -> horas | listas + edicoes em paralelo | ate tres awaits sequenciais removidos |

## 5. DIVIDA ESTRUTURAL REMANESCENTE

- `renderTrainingProgramRecordPage()` ainda busca template apos detalhe/opcoes porque a escolha depende de `tipo_treinamento_id` e `aeronave_modelo`. Remover esse encadeamento sem endpoint agregado seria tuning especulativo.
- O build continua sendo copia estatica sem bundler/tree-shaking. Esta frente reduziu startup eager por import dinamico, nao reabriu o sistema de build.
- O fallback de opcoes por base em tripulante continua por compat residual: ele deve morrer quando bases historicas/inativas estiverem normalizadas no contrato principal de opcoes.

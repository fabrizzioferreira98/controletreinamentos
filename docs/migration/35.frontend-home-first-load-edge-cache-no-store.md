# Bloco executado

Correcao minima da entrada SPA para eliminar dependencia de reload manual na home apos publicacao de frontend novo.

## Contexto

Sintoma observado em campo: a home (`/`) podia abrir com frontend antigo/parcial no primeiro acesso e so estabilizar no frontend novo apos reload manual.

## Causa raiz

- O build frontend usa versao em query string nos assets (`?v=...`) e essa versao e referenciada por `index.html`.
- O `Caddyfile` servia `index.html` sem politica explicita de `no-store`, permitindo reuso de HTML stale na entrada.
- Quando o navegador reutiliza `index.html` stale, ele carrega o grafo de assets antigo daquela versao.
- O reload manual revalida a entrada e passa a trazer o `index.html` atualizado, que aponta para os assets novos.

Owner da falha: edge delivery (`ops/windows/caddy/Caddyfile.example`), nao bootstrap/router/guard/sessao.

## Mudanca aplicada

- `@frontend_entry` passou a incluir `/index.html` alem de `/`, `/login`, `/logout`.
- O bloco de entrada recebeu headers:
  - `Cache-Control: no-store, no-cache, must-revalidate, max-age=0`
  - `Pragma: no-cache`
  - `Expires: 0`
- `/index.html` foi removido de `@frontend_assets` para nao cair fora do contrato de entrada.
- Aplicado nas duas versoes operacionais do Caddyfile (`app.seudominio.com` e `teste.seudominio.com`).

## Prova material

- Fluxo de boot SPA:
  - `frontend/src/app/bootstrap.js` faz `refreshSession` antes de resolver/renderizar rota.
  - `frontend/src/app/guards.js` consome permissoes da sessao ja carregada.
- Build e versionamento:
  - `frontend/scripts/build_frontend.py` aplica `?v=<timestamp>` em `.html` e `.js`.
- Borda corrigida:
  - `ops/windows/caddy/Caddyfile.example` agora marca a entrada HTML como `no-store`.

## Validacao

- Teste de contrato adicionado: `tests/contract/test_frontend_build_runtime_env_boundaries.py::test_frontend_edge_entry_html_cache_contract_in_caddy`.
- O teste trava regressao de cache na entrada SPA.
